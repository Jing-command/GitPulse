## 二、参考项目分析

### 2.1 changelog-generator

**项目地址**：https://github.com/metcalfc/changelog-generator

#### 2.1.1 项目概述

changelog-generator 是一个基于 GitHub Actions 的开源项目，能够生成两个 Git 引用之间的 Markdown 格式变更日志。该项目以简洁性和易用性为核心设计理念。

#### 2.1.2 核心功能

| 功能 | 说明 |
|------|------|
| 引用对比 | 支持任意两个 Git 引用之间的变更对比 |
| 自动化集成 | 基于 GitHub Actions，无需本地安装 |
| Markdown 输出 | 生成标准 Markdown 格式的变更日志 |
| 配置简单 | 仅需 GitHub Token 和引用参数即可运行 |

#### 2.1.3 技术架构

```
changelog-generator/
├── action.yml              # GitHub Action 配置
├── entrypoint.sh           # 入口脚本
├── src/
│   └── main.sh             # 核心逻辑
└── README.md
```

**输入参数**：
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `token` | string | 是 | GitHub Token |
| `headRef` | string | 否 | 头部引用（默认当前 SHA） |
| `baseRef` | string | 否 | 基础引用（默认最新发布标签） |
| `reverse` | boolean | 否 | 是否反转输出顺序 |
| `fetchAll` | boolean | 否 | 是否拉取所有分支和标签 |

**输出**：
- Markdown 格式的变更日志

#### 2.1.4 可借鉴之处

| 借鉴点 | 说明 | 应用场景 |
|--------|------|----------|
| 简洁的输入参数设计 | 仅需必要参数即可运行 | GitPulse CLI 命令设计 |
| GitHub Actions 集成 | 原生支持 CI/CD | GitPulse GitHub Action 开发 |
| Git 引用对比逻辑 | 支持任意引用范围 | GitPulse Diff 分析器 |
| Markdown 输出格式 | 标准化的输出结构 | GitPulse 内容生成器 |

#### 2.1.5 局限性

| 局限 | 说明 | GitPulse 改进方向 |
|------|------|-------------------|
| 无 AI 内容生成 | 仅输出 commit 列表 | 集成 AI 生成多种风格文案 |
| 无代码分析 | 不分析具体代码变更 | 实现 AST 分析，理解代码意图 |
| 单一输出格式 | 仅 Markdown | 支持多格式输出 |
| 无上下文感知 | 不读取项目配置 | 实现上下文感知系统 |
| 无文档同步 | 不自动更新文档站点 | 实现文档同步器 |

## 三、核心技术方案调研

### 3.1 Git Diff 解析方案

#### 3.1.1 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| simple-git | 轻量级、API 友好、纯 JS | 功能相对简单 | Node.js 项目 |
| nodegit | 功能强大、性能好 | 编译复杂、体积大 | 大型项目 |
| isomorphic-git | 纯 JS、跨平台 | 性能一般 | 浏览器环境 |
| 原生 git 命令 | 功能完整、性能最佳 | 需要系统依赖 | 服务端环境 |

**推荐方案**：`simple-git`

**理由**：
- 轻量级，安装简单
- API 设计友好，易于封装
- 社区活跃，维护良好
- 满足 GitPulse 所有需求

#### 3.1.2 simple-git 核心用法

```typescript
import simpleGit from 'simple-git';

const git = simpleGit('/path/to/repo');

// 获取 commit 列表
const logs = await git.log({ from: 'HEAD~10', to: 'HEAD' });

// 获取 diff
const diff = await git.diff(['HEAD~1', 'HEAD']);

// 获取指定 commit 的 diff
const show = await git.show(['--stat', 'a1b2c3d']);
```

### 3.2 AST 分析方案

#### 3.2.1 JavaScript/TypeScript 分析

| 工具 | 说明 | 优点 |
|------|------|------|
| @babel/parser | Babel 解析器 | 支持最新语法、插件丰富 |
| @typescript-eslint/parser | TypeScript 解析器 | 原生 TS 支持 |
| acorn | 轻量级解析器 | 体积小、速度快 |

**推荐方案**：`@babel/parser` + `@babel/traverse`

```typescript
import { parse } from '@babel/parser';
import traverse from '@babel/traverse';

const ast = parse(code, {
  sourceType: 'module',
  plugins: ['typescript', 'jsx'],
});

traverse(ast, {
  FunctionDeclaration(path) {
    console.log('Function:', path.node.id.name);
  },
});
```

#### 3.2.2 Python 分析

| 工具 | 说明 | 优点 |
|------|------|------|
| @python-parser/parser | JS 实现的 Python 解析器 | 纯 JS、无需 Python 环境 |
| pyright | 微软 Python 静态分析 | 功能强大 |
| tree-sitter-python | Tree-sitter Python 语法 | 性能优秀 |

**推荐方案**：`tree-sitter-python`

**理由**：
- 性能优秀
- 支持增量解析
- 统一的 Tree-sitter API

### 3.3 AI 服务集成方案

#### 3.3.1 服务商对比

| 服务商 | 模型 | 优点 | 缺点 |
|--------|------|------|------|
| OpenAI | GPT-4 | 质量高、API 稳定 | 成本较高 |
| Anthropic | Claude 3 | 长文本处理强 | 国内访问受限 |
| 本地模型 | Ollama | 无 API 成本 | 需要本地 GPU |

**推荐方案**：多服务商支持 + 降级策略

```typescript
interface AIProvider {
  name: string;
  generate(prompt: string): Promise<string>;
}

class AIService {
  private providers: AIProvider[];
  private currentProvider: number = 0;

  async generate(prompt: string): Promise<string> {
    for (let i = this.currentProvider; i < this.providers.length; i++) {
      try {
        return await this.providers[i].generate(prompt);
      } catch (error) {
        // 降级到下一个服务商
        this.currentProvider = i + 1;
      }
    }
    throw new Error('All AI providers failed');
  }
}
```

## 四、数据库方案调研

### 4.1 数据库选型

| 数据库 | 优点 | 缺点 | 适用场景 |
|--------|------|------|----------|
| PostgreSQL | 功能强大、扩展性好 | 部署复杂 | 大型项目 |
| MySQL | 生态成熟、社区活跃 | 功能相对简单 | 中型项目 |
| SQLite | 轻量级、无需部署 | 并发性能差 | 小型项目 |

**推荐方案**：`PostgreSQL`（主）+ `MySQL`（兼容）

**理由**：
- 支持 JSON 类型，适合存储配置和元数据
- 全文搜索功能
- Prisma ORM 支持良好

### 4.2 ORM 选型

| ORM | 优点 | 缺点 |
|-----|------|------|
| Prisma | 类型安全、迁移管理方便 | 查询复杂度限制 |
| TypeORM | 功能全面、装饰器语法 | 配置复杂 |
| Sequelize | 成熟稳定 | TypeScript 支持一般 |

**推荐方案**：`Prisma`

**理由**：
- 类型安全，开发体验好
- 迁移管理方便
- 支持多数据库
- 查询构建器直观

## 五、前端方案调研

### 5.1 框架选型

| 框架 | 优点 | 缺点 |
|------|------|------|
| React 18 | 生态丰富、团队熟悉 | 学习曲线 |
| Vue 3 | 上手简单、性能好 | 生态相对小 |
| Svelte | 编译时优化、体积小 | 生态较小 |

**推荐方案**：`React 18`

### 5.2 状态管理

| 方案 | 优点 | 缺点 |
|------|------|------|
| React Query | 服务端状态管理优秀 | 仅限数据获取 |
| Zustand | 轻量级、简单 | 功能相对简单 |
| Redux Toolkit | 功能全面 | 模板代码多 |

**推荐方案**：`React Query`（服务端状态）+ `Zustand`（客户端状态）

### 5.3 UI 组件库

| 组件库 | 优点 | 缺点 |
|--------|------|------|
| Ant Design | 组件丰富、文档完善 | 体积较大 |
| shadcn/ui | 可定制性强、现代设计 | 组件相对少 |
| Chakra UI | 可访问性好、主题系统强 | 学习成本 |

**推荐方案**：`shadcn/ui` + `Tailwind CSS`

**理由**：
- 组件代码直接复制，可完全定制
- 现代设计风格
- 与 Tailwind CSS 完美集成

## 六、关键技术决策总结

| 技术领域 | 选型 | 理由 |
|----------|------|------|
| 运行时 | Node.js 18+ | 生态丰富、Git 工具链完善 |
| 开发语言 | TypeScript 5.x | 类型安全、开发体验好 |
| Git 操作 | simple-git | 轻量级、API 友好 |
| JS/TS AST | @babel/parser | 支持最新语法 |
| Python AST | tree-sitter-python | 性能优秀 |
| AI 服务 | 多服务商支持 | 灵活降级 |
| 数据库 | PostgreSQL | 功能强大、JSON 支持 |
| ORM | Prisma | 类型安全、迁移方便 |
| 前端框架 | React 18 | 生态丰富 |
| 状态管理 | React Query + Zustand | 职责分离 |
| UI 组件 | shadcn/ui + Tailwind | 可定制、现代设计 |
| 文档站点 | VitePress | 性能优秀、配置简单 |

## 七、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| AI API 成本过高 | 高 | 本地模型支持、缓存机制 |
| AST 解析复杂度 | 中 | 先支持主流语言，逐步扩展 |
| 大型仓库性能 | 中 | 增量分析、分批处理 |
| 多语言翻译质量 | 中 | 高质量模型、人工校对 |
